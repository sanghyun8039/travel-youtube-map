분석 파이프라인 — 3가지 접근 방식

영상 URL 입력부터 지도 + 블로그 생성까지의 아키텍처

A

스트리밍 단일 파이프라인 ★ 추천

URL 입력 → 서버가 순서대로 처리하며 결과를 실시간 스트리밍으로 프론트에 전달

장점

  • 진행 상황 실시간 표시 (자막 추출 중… 키프레임 분석 중…)
  • 코드 단순, MVP에 최적
  • Next.js Server-Sent Events로 구현 용이

단점

  • 긴 영상(1시간+)은 응답 대기 길어질 수 있음
  • 중간에 끊기면 재시작 필요
B

백그라운드 잡 + 폴링

URL 제출 → Job ID 발급 → 서버 백그라운드 처리 → 프론트가 완료 시 결과 수신

장점

  • 긴 영상도 안정적 처리
  • 결과 캐싱/재사용 용이

단점

  • 큐/DB 추가 필요 (복잡도 증가)
  • Vercel 서버리스와 궁합 안 좋음
C

단계별 확인 파이프라인

각 단계(자막 → 키프레임 → LLM)마다 중간 결과를 사용자가 확인·수정 후 다음 단계 진행

장점

  • 오류를 중간에 수정 가능
  • 장소 추출 결과 직접 편집 가능

단점

  • 클릭 단계 많아 UX 복잡
  • 블로그 자동화 흐름 끊김

공통 토큰 최적화 전략

① YouTube Transcript
자막 있으면 무료 즉시 사용 → 토큰 0
② Whisper fallback
자막 없을 때만 음성→텍스트 변환
③ 키프레임 샘플링
30초마다 1장 or 장면 전환 감지 시만

클릭으로 선택하거나 터미널에 의견을 알려주세요
